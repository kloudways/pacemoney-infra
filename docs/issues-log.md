# Issues log

All issues encountered during Phases 1-8 of the Pace Money build.

| Issue | Phase | Symptom | Root cause | Fix | Prevention |
|-------|-------|---------|-----------|-----|-----------|
| `python3-venv` missing on Jenkins | 3 | `python3 -m venv .venv` failed during Unit Tests stage | Ansible common role did not install `python3-venv` or `python3-pip` | Added both packages to `ansible/roles/common/tasks/main.yml` and installed live via SSH | Include the full Python toolchain in the base Ansible role |
| Jenkins `ec2:DescribeAvailabilityZones` denied | 5 | kops failed to export kubeconfig during Helm Deploy | Jenkins IAM inline policy did not include `ec2:DescribeAvailabilityZones` | Added the permission and applied via `terraform apply -target=aws_iam_role_policy.jenkins` | Provision the full kops read-only permission set upfront |
| Jenkins ELB and Auto Scaling permissions denied | 5 | kops could not read cluster state; multiple AccessDenied errors for `elasticloadbalancing:Describe*` and `autoscaling:Describe*` | Jenkins IAM policy missing ELB v2 and Auto Scaling read permissions required by kops | Added all missing Describe* permissions in one batch and applied via Terraform | Include the complete set of kops-required read permissions in the initial IAM policy |
| RDS not reachable from kops nodes | 5 | Pods crashed silently (exit code 137, empty logs); liveness probe reported `connection refused` | The RDS security group allowed port 5432 only from `app_node_sg`. kops nodes are not assigned `app_node_sg` (kops manages its own security groups), so the TCP connection to RDS hung before uvicorn could start | Added a second ingress rule to `pacemoney-rds-sg` allowing port 5432 from the VPC CIDR (10.1.0.0/16) | Ensure security group rules cover all compute that needs database access, including kops-managed node security groups |
| kops state bucket blocks `terraform destroy` | 5 | `terraform destroy` failed with `BucketNotEmpty: The bucket you tried to delete is not empty. You must delete all versions in the bucket` | The kops state bucket has versioning enabled; `terraform destroy` cannot delete a versioned bucket with objects | Removed the bucket from Terraform state with `terraform state rm aws_s3_bucket.kops_state` and left it in AWS (it costs less than $0.01/month and is reused on the next startup) | Document this as an expected teardown step |
| ECR repository blocks `terraform destroy` | 5 | `terraform destroy` failed with `RepositoryNotEmptyException` | The ECR repository still contained images pushed during the pipeline run | Force-deleted the repository: `aws ecr delete-repository --repository-name pacemoney --force --region eu-west-2` | Add `force_delete = true` to the `aws_ecr_repository` resource, or document the CLI command in the teardown runbook |
| kops `cluster.yaml` has stale VPC/subnet IDs after restart | 6 | `kops update cluster` retried for 10 minutes with `InvalidVpcID.NotFound` | `cluster.yaml` had VPC and subnet IDs hardcoded from the previous session. Every `terraform destroy` + `terraform apply` creates a new VPC with new IDs. | Updated IDs manually for the immediate fix; then automated by adding a `local_file` Terraform resource that renders `cluster.yaml` from a template using live Terraform outputs on every apply | Never hardcode AWS resource IDs in kops config; always generate from Terraform outputs |
| `kops create` used wrong instance group file name | 6 | `kops create -f terraform/kops/master.yaml` failed with file not found | Runbook referenced `master.yaml` and `nodes.yaml` but the repo only has `instancegroups.yaml` | Ran `kops create -f terraform/kops/instancegroups.yaml` instead; updated runbook | Keep runbook commands in sync with actual file names |
| Helm deploy failed with `no matches for kind ServiceMonitor` | 6 | Helm Deploy stage failed; all subsequent stages skipped | `kube-prometheus-stack` (which installs the `ServiceMonitor` CRD) had not been deployed yet when the pipeline first ran | Deployed `kube-prometheus-stack` via Helm from the local machine first, then re-ran the pipeline | Always deploy the monitoring stack before the first app deployment in a fresh cluster |
| Helm API discovery cache stale on Jenkins | 6 | Helm Deploy still failed with `no matches for kind ServiceMonitor` after monitoring stack was installed | Helm caches API group discovery locally; the Jenkins server had a stale cache that did not include the new CRD | Deleted the Helm cache on Jenkins: `sudo -u jenkins rm -rf /var/lib/jenkins/.cache/helm/` | Clear the Helm cache after installing CRDs from a separate Helm release, or add a cache-clear step to the Jenkinsfile |
| `db_password` prompted on every `terraform apply` | 6 | Manual password entry required on every startup | `db_password` was declared as a variable with no default and not stored in `terraform.tfvars` | Added `db_password` to `terraform.tfvars` (gitignored) | Always add sensitive-but-stable variables to the gitignored `terraform.tfvars` at project start |
| ESO `SecretStore` and `ExternalSecret` used wrong API version | 8 | ArgoCD sync failed: "The Kubernetes API could not find version `v1beta1` of `external-secrets.io/SecretStore`" | The Helm templates declared `apiVersion: external-secrets.io/v1beta1` but the installed ESO version promotes both resources to `v1` | Changed `apiVersion` to `external-secrets.io/v1` in both `secretstore.yaml` and `externalsecret.yaml` | Always verify the API version supported by the installed ESO release before writing templates |
| ESO pods could not reach EC2 IMDS from inside containers | 8 | `SecretStore` status: `InvalidProviderConfig — no EC2 IMDS role found, context deadline exceeded` | The EC2 IMDS hop limit was 1 (the default). This is sufficient for host processes but containers in pods require a hop limit of at least 2 | Ran `aws ec2 modify-instance-metadata-options --http-put-response-hop-limit 2` on both worker nodes | Add the hop limit step to the runbook immediately after `kops update cluster --yes` |
| kops-managed node IAM role missing Secrets Manager permission | 8 | ESO `AccessDeniedException` for `secretsmanager:GetSecretValue` on role `nodes.pacemoney.k8s.local` | Terraform manages a role named `pacemoney-kops-node` with the Secrets Manager policy, but kops creates its own role `nodes.pacemoney.k8s.local` and attaches it to the nodes; the Terraform-managed role is not used by the nodes | Added an inline policy directly to `nodes.pacemoney.k8s.local` via `aws iam put-role-policy` | Add a Terraform `aws_iam_role_policy` resource targeting `nodes.pacemoney.k8s.local` or use kops `additionalPolicies` in `cluster.yaml` to make this persistent |
| Jenkins user not in docker group | 9 | Build #1 failed — `permission denied while trying to connect to Docker daemon socket at unix:///var/run/docker.sock` | The Ansible jenkins role did not add the `jenkins` user to the `docker` group | `sudo usermod -aG docker jenkins && sudo systemctl restart jenkins` on the Jenkins host | Add `usermod -aG docker jenkins` to the Ansible jenkins role tasks so it is applied on every provision |
| SSH timeout to Jenkins after daily restart | 9 | `ssh: connect to host jenkins.kloudways.com port 22: Operation timed out` | Jenkins EC2 instance IP changes on every start; the security group inbound SSH rule was locked to the previous session's IP | Updated the security group inbound rule to the current public IP via AWS console or CLI | Use an Elastic IP for Jenkins, or widen the SSH CIDR; update the security group IP as part of the daily startup checklist |
| SSH host key verification failed | 9 | `WARNING: REMOTE HOST IDENTIFICATION HAS CHANGED!` when SSHing to Jenkins | A new EC2 instance has a different host key from the previous session | `ssh-keyscan -H jenkins.kloudways.com >> ~/.ssh/known_hosts` | Expected on every fresh EC2 instance; add the scan step to the runbook |
| `cluster.yaml` missing after teardown | 9 | `kops create -f terraform/kops/cluster.yaml` failed with file not found | `cluster.yaml` is generated by a Terraform `local_file` resource; after `terraform destroy` the file is deleted and is only regenerated on `terraform apply` | Run `terraform apply -auto-approve -target=local_file.kops_cluster_config` to regenerate the file before running kops commands | Document this as a required step in the startup runbook; the file is not committed to git |
